A **world** is the foundational unit of organization in the Worlds Platform. It provides an isolated container for an agent's relationships, history, and facts.

<Note>
  **The problem: Why AI forgets**

  Traditional AI can only "remember" what's in its current chat window. Once that window fills up or the chat ends, it loses the thread. This makes it hard for agents to handle complex jobs that take more than one session.

  **The solution: Linked memory**

  Instead of hoping the AI finds the right data in a massive pile, Worlds creates a structured map (a knowledge graph). By linking related facts together, your agent can always follow the logical path from one piece of information to another, no matter how much data you add.
</Note>

## Goals

| Goal | Description | Constraint |
| :--- | :--- | :--- |
| **Isolation** | Prevent data leakage between agents. | One RDF dataset per world. |
| **Portability** | Maintain memory across model swaps. | Decoupled from LLM provider. |
| **Fusion** | Hybrid neural/symbolic reasoning. | Requires both vector + SPARQL. |

## Architecture

A world functions as a **neuro-symbolic memory container**. It pairs an isolated RDF dataset with a high-performance vector search index.

### Neural layer

The neural layer uses vector embeddings to manage semantic similarity. It allows the agent to navigate the world through natural language queries. High-dimensional embeddings (1536-dim) power semantic search over stored text chunks.

### Symbolic layer

The symbolic layer uses RDF and SPARQL for deterministic facts. It provides deterministic validation, reducing the risk of hallucinations inherent in standalone LLMs.

## RDF triples as facts

All knowledge in a world is stored as **RDF triples** — atomic statements of the form:

```
subject  predicate  object
```

For example:

```turtle
<http://example.com/ethan>   <http://schema.org/relatedTo>   <http://example.com/gregory> .
<http://example.com/gregory> <http://schema.org/givenName>   "Gregory" .
```

These triples form a directed graph. When an agent queries the world, it traverses this graph using SPARQL — enabling multi-hop reasoning that vector search cannot provide.

## Knowledge graphs vs. vector stores

| Capability | Vector store | World (knowledge graph) |
| :--- | :--- | :--- |
| Semantic similarity | Yes | Yes (via hybrid search) |
| Structural relationships | No | Yes (RDF + SPARQL) |
| Multi-hop traversal | No | Yes |
| Verifiable audit trail | No | Yes |
| Mutable facts | Requires re-index | Atomic SPARQL update |

## Malleable knowledge

Worlds are not static indexes. You can mutate specific facts without re-indexing the entire graph:

```typescript
// Update a single fact
await sdk.worlds.sparql(
  worldId,
  `
  DELETE { <http://example.com/ethan> <http://schema.org/jobTitle> "Engineer" . }
  INSERT { <http://example.com/ethan> <http://schema.org/jobTitle> "Senior Engineer" . }
  WHERE  { <http://example.com/ethan> <http://schema.org/jobTitle> ?oldTitle . }
`,
);
```

This `rdf-patch` operation atomically replaces a single triple without touching any other data in the world.

## Audit trails

Worlds eliminates "black box" behavior. Every retrieved fact has a verifiable audit trail in the RDF graph. If an agent makes a claim, the symbolic layer proves it.

This is critical for high-stakes domains like medicine, law, and compliance, where agents must provide a perfect trace of their reasoning.

## Accessing a world

You can interact with a world through the SDK:

```typescript
import { WorldsSdk } from "@wazoo/worlds-sdk";

const sdk = new WorldsSdk({
  baseUrl: "http://localhost:8000",
  apiKey: "your-api-key",
});

// Create a world
const world = await sdk.worlds.create({
  slug: "agent-memory",
  label: "Agent Memory",
});

// Get a world
const existing = await sdk.worlds.get("agent-memory");

// List all worlds
const all = await sdk.worlds.list({ page: 1, pageSize: 20 });
```
